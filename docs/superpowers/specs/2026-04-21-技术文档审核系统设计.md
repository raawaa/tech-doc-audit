# 技术文档审核系统设计文档

## 1. 项目概述

### 1.1 项目背景

构建一个技术文档审核应用，用户可上传技术文档（如招标文件的技术要求），系统通过知识库（技术标准、规范）对比审核上传的文档，生成详细的审核报告。

### 1.2 核心需求

| 需求项 | 具体描述 |
|--------|----------|
| **审核类型** | 综合审核：合规性、完整性、一致性三种类型 |
| **知识库来源** | 内部文档库、公开标准库、结构化知识库等多种来源 |
| **文档格式** | 支持 PDF、Word（doc/docx）格式 |
| **审核结果** | 问题清单：包含问题位置（原文）、问题类型、依据的标准条文 |
| **部署方式** | 本地部署，企业级使用 |
| **用户规模** | 组织内部使用（几个到几十人） |

### 1.3 技术约束

- 必须本地部署，不依赖外部云服务
- 无现有基础设施，需要从头搭建
- LLM 模型待定，倾向于本地部署开源模型

---

## 2. 技术选型

### 2.1 技术方案对比

| 技术 | 定位 | 本地部署 | 特点 |
|------|------|----------|------|
| **Dify** | LLM 应用开发平台 | ✅ 支持 | 可视化工作流、RAG 管道、多模型支持 |
| **PageIndex** | 无向量 RAG 检索引擎 | ✅ 支持（开源） | 层次树索引、推理式检索、高可解释性 |
| **Sirchmunk** | 无索引知识检索引擎 | ✅ 支持 | 无需预索引、自进化知识库 |

### 2.2 选型决策

采用 **Dify + PageIndex 组合架构**：

- **PageIndex** 作为检索引擎：推理式检索、可解释性强、适合结构化文档、无需向量数据库
- **Dify** 作为应用平台：成熟的工作流编排、快速构建企业应用、本地部署支持
- **不选 Sirchmunk**：更适合代码库/动态数据检索，对结构化文档检索不如 PageIndex 精确

## 4. 核心功能模块

### 4.1 知识库管理模块

| 功能 | 描述 |
|------|------|
| **知识库导入** | 支持批量导入标准规范文档（PDF/Word），自动生成 PageIndex 树索引 |
| **知识库分类** | 按类型分类管理（国家标准、行业标准、企业规范等） |
| **知识库更新** | 支持增量更新、版本管理 |
| **索引维护** | 定期检查索引状态，支持重建索引 |

**数据结构**：
```
知识库/
├── 国家标准/
│   ├── GB-XXXX-2023.pdf        → pageindex_tree.json
│   └── GB-YYYY-2022.pdf        → pageindex_tree.json
├── 行业标准/
│   ├── JB-XXXX-2021.pdf        → pageindex_tree.json
│   └── ...
└── 企业规范/
    └── ...
```

### 4.2 文档上传与解析模块

| 功能 | 描述 |
|------|------|
| **文档上传** | 用户上传待审核文档（PDF/Word），支持单文件或多文件 |
| **文档解析** | 提取文档文本内容，保留章节结构信息 |
| **结构识别** | 自动识别文档目录结构（章节、条款编号） |
| **临时索引** | 为上传文档生成 PageIndex 临时索引，用于后续检索 |

**解析流程**：
```
上传文档 → 文件校验 → 文本提取 → 结构识别 → PageIndex索引 → 存储
```

### 4.3 审核分析模块（核心）

**审核流程**：
```
┌─────────────────────────────────────────────────────────────┐
│                     审核工作流                               │
│                                                             │
│  ① 文档结构分析                                              │
│     - 解析文档章节结构                                        │
│     - 提取关键技术要求条目                                    │
│                                                             │
│  ② 逐条审核                                                  │
│     - 对每个技术要求条目：                                    │
│       • PageIndex 检索知识库相关标准条文                      │
│       • LLM 对比分析：是否符合、是否有遗漏、是否一致           │
│       • 生成审核结果                                          │
│                                                             │
│  ③ 结果汇总                                                  │
│     - 汇总所有问题                                           │
│     - 分类整理：合规性问题 / 完整性问题 / 一致性问题          │
│     - 生成最终审核报告                                        │
└─────────────────────────────────────────────────────────────┘
```

**审核结果数据结构**：
```json
{
  "document_id": "xxx",
  "audit_time": "2026-04-21 10:00:00",
  "summary": {
    "total_issues": 15,
    "compliance_issues": 8,
    "completeness_issues": 4,
    "consistency_issues": 3
  },
  "issues": [
    {
      "id": 1,
      "type": "compliance",
      "location": {
        "section": "第三章 技术要求",
        "clause": "3.2.1",
        "page": 12,
        "original_text": "设备防护等级应达到IP54..."
      },
      "standard_reference": {
        "standard": "GB/T 4208-2017",
        "clause": "5.2",
        "requirement": "户外设备防护等级应达到IP65及以上"
      },
      "issue_description": "防护等级要求不符合国家标准...",
      "severity": "high"
    }
  ]
}
```

### 5.2 PageIndex 检索策略

| 步骤 | 操作 | 说明 |
|------|------|------|
| **索引生成** | 对知识库文档生成树索引 | 预处理阶段完成，持久化存储 |
| **检索请求** | 以技术条目为查询输入 | 例如："设备防护等级要求" |
| **树搜索** | PageIndex 推理式检索 | LLM 在树结构中导航，定位相关章节 |
| **内容提取** | 返回匹配的标准条文 | 包含：标准名称、章节、条款号、原文 |

**检索参数建议**：
- `max-pages-per-node`: 5-10（保持细粒度）
- `max-tokens-per-node`: 15000-20000
- 检索模式：使用推理式检索（而非简单的关键词匹配）

### 5.3 LLM 调用策略

| 调用场景 | 模型建议 | 说明 |
|----------|----------|------|
| **文档结构分析** | 中等规模模型 (Qwen-14B/GLM-4-9B) | 提取章节、条目，不需要超强推理 |
| **PageIndex 推理检索** | 中等规模模型 | 树导航推理，中等模型足够 |
| **对比分析** | 较强模型 (Qwen-72B/GLM-4) | 需要深度理解标准条文和原文，判断合规性 |
| **报告生成** | 中等规模模型 | 格式化输出，相对简单 |

**优化策略**：
- 并行处理：多个条目可并行审核（控制在 3-5 个并发）
- 缓存：相同条目的检索结果可缓存
- 批处理：批量提交 LLM 请求减少调用次数

### 5.4 错误处理

| 错误类型 | 处理策略 |
|----------|----------|
| **文档解析失败** | 提示用户检查文档格式，记录错误日志 |
| **PageIndex 索引失败** | 自动重试 2 次，失败后人工介入 |
| **LLM 调用超时** | 设置合理超时（60-120秒），超时后重试 |
| **知识库检索无结果** | 标记为"需人工复核"，不判定为合规/不合规 |
| **并发限制** | 使用队列控制并发，避免资源耗尽 |

## 6. API 设计

### 6.1 端点列表

**基础路径**: `/api/v1`

| 模块 | 端点 | 方法 | 功能 |
|------|------|------|------|
| **知识库管理** | `/knowledge-bases` | GET | 获取知识库列表 |
| | `/knowledge-bases` | POST | 创建知识库 |
| | `/knowledge-bases/{id}/documents` | POST | 导入文档到知识库 |
| | `/knowledge-bases/{id}/documents/{docId}` | DELETE | 删除知识库文档 |
| | `/knowledge-bases/{id}/reindex` | POST | 重建知识库索引 |
| **文档审核** | `/documents` | POST | 上传待审核文档 |
| | `/documents/{id}` | GET | 获取文档信息 |
| | `/documents/{id}/audit` | POST | 启动审核任务 |
| | `/documents/{id}/audit/status` | GET | 获取审核进度 |
| | `/documents/{id}/audit/result` | GET | 获取审核结果 |
| | `/documents/{id}/audit/report` | GET | 导出审核报告（PDF/Word） |
| **用户管理** | `/users` | GET | 用户列表（管理员） |
| | `/users/me` | GET | 当前用户信息 |
| | `/auth/login` | POST | 登录 |
| | `/auth/logout` | POST | 登出 |
| **系统管理** | `/system/status` | GET | 系统状态（健康检查） |
| | `/system/models` | GET | 可用 LLM 模型列表 |

### 6.2 核心接口详情

#### 上传文档

```http
POST /api/v1/documents
Content-Type: multipart/form-data

参数：
- file: 文档文件（PDF/Word）
- name: 文档名称（可选）
- knowledge_base_ids: 知识库ID列表（可选）

响应：
{
  "document_id": "doc_001",
  "status": "uploaded",
  "message": "文档上传成功，等待审核"
}
```

#### 启动审核任务

```http
POST /api/v1/documents/{id}/audit
Content-Type: application/json

参数：
{
  "knowledge_base_ids": ["kb_001", "kb_002"],
  "audit_types": ["compliance", "completeness", "consistency"],
  "priority": "normal"
}

响应：
{
  "audit_id": "audit_001",
  "status": "processing",
  "estimated_time": 120,
  "progress_url": "/api/v1/documents/doc_001/audit/status"
}
```

#### 获取审核进度

```http
GET /api/v1/documents/{id}/audit/status

响应：
{
  "audit_id": "audit_001",
  "status": "processing",
  "progress": {
    "total_items": 50,
    "processed_items": 25,
    "current_stage": "audit_analysis",
    "percentage": 50
  },
  "started_at": "2026-04-21 10:00:00",
  "estimated_completion": "2026-04-21 10:02:00"
}
```

### 6.3 通信方式

| 场景 | 方式 | 说明 |
|------|------|------|
| **文档上传** | HTTP POST + multipart | 文件上传，同步返回文档ID |
| **审核进度** | WebSocket / SSE | 实时推送审核进度 |
| **审核结果** | HTTP GET | 审核完成后获取完整结果 |
| **报告导出** | HTTP GET + 流式下载 | 大文件流式下载 |
