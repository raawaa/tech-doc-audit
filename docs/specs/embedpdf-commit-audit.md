# Commit audit: can the `commit()` calls in `PdfViewer.tsx` be removed? (issue #81)

> Generated by background research agent for issue #81 (the reconcile ticket for issue #75
> and issue #79). Sources cross-checked: `@embedpdf/plugin-annotation` v2.14.4 bundled JS
> (`frontend/node_modules/@embedpdf/plugin-annotation/dist/index.js`), the upstream GitHub
> primary source (`https://github.com/embedpdf/embed-pdf-viewer`, branch `main`,
> `packages/plugin-annotation/src/lib/annotation-plugin.ts`), the per-package CHANGELOG,
> `frontend/src/pages/PdfViewer.tsx`, `docs/specs/embedpdf-annotation-async-contract.md`
> (predecessor spec from #75/#76), `docs/specs/pdf-viewer-url-contract-verification.md`
> (predecessor verification from #74), `docs/adr/0006-pdf-viewer-embedpdf-dropin.md`
> (ADR that introduced the explicit `commit()` calls as pitfall #3), and the empirical
> finding from #79 that disabling both call sites produced no observable change.

## Verdict

**Decision: REMOVE** — the explicit `commit()` calls at
`frontend/src/pages/PdfViewer.tsx:497` and `:543` are structurally redundant under the
currently pinned `@embedpdf/plugin-annotation@2.14.4` and its default `autoCommit: true`
config. They can be removed without changing observable behavior, on either code path
(`handleReady.onLayoutReady` or the §5 fallback useEffect). ADR-0006 pitfall #3 ("after
`importAnnotations(...)`, always call `commit()` explicitly") should be rewritten in the
follow-up ADR redline (issue #82), not in this issue.

No RETAIN edge case is required to pass the four-timing-window reload test. The spec §5.3
sentence that follows in §6 below records what the spec needs to say, in case future
reviewers want the one-liner that supersedes the prior wording.

## Section-by-section findings

### §1 Structural path of `importAnnotations` — IT DOES call `commit()`

- **Status**: CORRECTS prior diagnosis from issue #75.
- Verified at `frontend/node_modules/@embedpdf/plugin-annotation/dist/index.js:5241-5269`
  (annotation-plugin.ts L976-L1002 upstream):
  ```
  importAnnotations(items, documentId) {
    const id = documentId ?? this.getActiveDocumentId();
    if (!this.isInitialLoadComplete.get(id)) {
      const queue = this.importQueue.get(id) || [];
      queue.push(...items);
      this.importQueue.set(id, queue);
      return;
    }
    this.processImportItems(id, items);
  }
  processImportItems(documentId, items) {
    const contexts = this.pendingContexts.get(documentId);
    if (!contexts) return;
    for (const item of items) {
      const { annotation, ctx } = item;
      const pageIndex = annotation.pageIndex;
      const id = annotation.id;
      this.dispatch(createAnnotation(documentId, pageIndex, annotation));
      if (ctx) contexts.set(id, ctx);
    }
    if (this.config.autoCommit !== false) this.commit(documentId);
  }
  ```
  **The last line of `processImportItems` (line 5268) is the dispositive fact**:
  `if (this.config.autoCommit !== false) this.commit(documentId);`. Under the default
  config (`autoCommit: true`, see §2), this fires a `commit()` on the same tick that
  `importAnnotations()` returns.
- The inline comment in `PdfViewer.tsx:495-496` ("import 路径不走那条 reducer",
  "autoCommit:true 只对 CREATE_ANNOTATION 路径生效") is **factually wrong** against the
  bundled JS. The auto-commit guard at `index.js:5268` is the import-path analog of
  the auto-commit guard at `index.js:5303` (the `createAnnotation` method). Both guard
  the same `commit()` entrypoint. The predecessor spec
  (`docs/specs/embedpdf-annotation-async-contract.md` §7 lines 461-462) flags this same
  inaccuracy and recommends the same correction; this audit re-confirms it against
  fresh read of the file.
- Cross-check against upstream GitHub primary source:
  `https://github.com/embedpdf/embed-pdf-viewer/blob/main/packages/plugin-annotation/src/lib/annotation-plugin.ts`
  lines 988-L1002: identical body. The auto-commit-on-import behavior is not a
  bug-fix-in-progress or a stale pinned bundle artifact — it is the live upstream
  behavior on `main`.

### §2 The `commitPendingChanges` reducer — reachable from both paths

- **Status**: ACCURATE (matches predecessor spec §3).
- Verified `COMMIT_PENDING_CHANGES` reducer at
  `frontend/node_modules/@embedpdf/plugin-annotation/dist/index.js:4558-4585`:
  ```
  case COMMIT_PENDING_CHANGES: {
    const { documentId, committedUids } = action.payload;
    const docState = state.documents[documentId];
    if (!docState) return state;
    const committedSet = new Set(committedUids);
    const cleaned = {};
    let stillHasPending = false;
    for (const [uid, ta] of Object.entries(docState.byUid)) {
      if (committedSet.has(uid)) {
        cleaned[uid] = {
          ...ta,
          commitState: ta.commitState === "dirty" || ta.commitState === "new" || ta.commitState === "moved" ? "synced" : ta.commitState
        };
      } else {
        cleaned[uid] = ta;
        if (ta.commitState === "new" || ta.commitState === "dirty" || ta.commitState === "moved" || ta.commitState === "deleted") {
          stillHasPending = true;
        }
      }
    }
    ...
  }
  ```
- This reducer is reached from **one and only one** producer:
  `executeCommitBatch` at `index.js:6754`:
  ```
  Task.allSettled(allTasks).wait(
    () => {
      this.emitCommitEvents(docId, pendingOps, contexts);
      ...
      this.dispatch(commitPendingChanges(docId, batch.committedUids));
      task.resolve(true);
    },
    ...
  );
  ```
  `executeCommitBatch` is invoked only from `commit()` at `index.js:6841`. **Every path
  that flips `commitState: 'new' → 'synced'` goes through `commit()`.** The auto-commit
  inside `processImportItems` (line 5268) and the manual `commit()` in PdfViewer.tsx
  are the same entrypoint.
- **Implication**: if `importAnnotations` is called and the auto-commit fires, the state
  reaches `'synced'` exactly as if the manual call had been made. The only observable
  difference would be **timing** (the manual call resolves synchronously after
  `acquireCommitLock` returns true; the auto-commit fires during the same call stack but
  `commit()` itself is async — see §3).

### §3 `autoCommit` snippet option — default `true` in the public config

- **Status**: NEW FINDING (extends predecessor spec §7, corrects prior claim).
- Verified the snippet factory default at
  `frontend/node_modules/@embedpdf/plugin-annotation/dist/index.js:12-19`:
  ```
  defaultConfig: {
    autoCommit: true,
    annotationAuthor: "Guest",
    deactivateToolAfterCreate: false,
    selectAfterCreate: true,
    autoOpenLinks: true
  }
  ```
- Verified the public type signature at
  `frontend/node_modules/@embedpdf/plugin-annotation/dist/lib/types.d.ts:139-161`:
  ```
  export interface AnnotationPluginConfig extends BasePluginConfig {
    ...
    /** When true (default), automatically commit the annotation changes into the PDF document. */
    autoCommit?: boolean;
    ...
  }
  ```
  Note JSDoc: "When true (default)...". This is the public, documented default. No
  `@deprecated` tag.
- PdfViewer.tsx path does NOT override `autoCommit`. The viewer is built via
  `@embedpdf/react-pdf-viewer` (verified version 2.14.4 at
  `frontend/node_modules/@embedpdf/react-pdf-viewer/package.json:3`). The dropin does
  not pass `autoCommit: false` (verified absence at
  `frontend/src/pages/PdfViewer.tsx:593-605` dropinConfig shape — no `annotation`
  plugin config field).
- **Auto-commit on `importAnnotations` path (line 5268) — present, fires when default is
  used**: `if (this.config.autoCommit !== false) this.commit(documentId);`
- **Auto-commit on `createAnnotation` path (line 5303) — present but with TRUTHY check
  (not `!== false`)**: `if (this.config.autoCommit) this.commit(docId);`
- The asymmetry between `!== false` (line 5268) and truthy check (line 5303) means the
  import path fires for both `undefined` and `true`, but silently degrades if
  `autoCommit === false`. **The PdfViewer.tsx path is the import path, so the relevant
  check is `!== false`.** Default `autoCommit: true` → guard passes → commit fires.
- **What `autoCommit: true` does**: auto-dispatches `commit()` immediately after the
  CREATION reducer path runs (line 5303) AND after the IMPORT path runs (line 5268).
  It does NOT directly toggle `commitState` to `'synced'`; that flip happens later in
  `executeCommitBatch` when the engine `createPageAnnotation` task resolves. So the
  `commit()` call is the trigger, but the `'synced'` transition is engine-tick-late
  (verified in predecessor spec §3 lines 122-235).

### §4 Pinned version & release history

- **Status**: ACCURATE.
- Verified pinned version at `frontend/package.json:12` (and lockfile):
  `"@embedpdf/plugin-annotation": "^2.14.4"`. Resolved at
  `frontend/node_modules/@embedpdf/plugin-annotation/package.json:3`:
  `"version": "2.14.4"`.
- The auto-commit-on-import behavior was **introduced in commit
  `5a3c6d38c6834fd703b9efe138240610034f30a0`** upstream (2025-09-17, "Add ability to
  bulk import annotations",
  `https://github.com/embedpdf/embed-pdf-viewer/commit/5a3c6d38c6834fd703b9efe138240610034f30a0`).
  This is the same commit that introduced `importAnnotations` itself, so the
  auto-commit and the public import API have always shipped together.
- **Per-package CHANGELOG**
  (`https://github.com/embedpdf/embed-pdf-viewer/blob/main/packages/plugin-annotation/CHANGELOG.md`):
  - v2.12.0 (2026-04-01, PR #569): introduced the symmetric import/export API with the
    unified `AnnotationTransferItem` type. No revert or breakage note.
  - v2.13.0: patch release (re-export patching utilities, PR #579). No relevant change.
  - v2.14.x: covers callout FreeText, lock helpers, Vue/Svelte renderer fixes. None
    touch the import/commit semantics.
- **Closest evidence gap**: the precise release tag for `5a3c6d3` is not directly
  marked in the CHANGELOG (the PR for bulk import came in via the bulk-import feature
  branch and was merged ahead of the next minor version). The behavior has been in
  every published version ≥ 2.12.0, which subsumes the pinned 2.14.4. **No gap in
  practice: for any version we'd pin to, the auto-commit is present.**

### §5 Future-version risk — observable signal exists

- **Status**: RISK IS LOW. A future revert would announce itself via:
  1. **Public JSDoc change** on `AnnotationPluginConfig.autoCommit`
     (`types.d.ts:147-148` upstream). If the JSDoc dropped "When true (default)..." and
     became "When true...", that's the breaking signal. Diff of upstream `types.ts` is
     the surveillance surface.
  2. **Per-package CHANGELOG** entry under `### Major Changes` or `### Minor Changes`
     with a PR link. The repo uses mandatory changesets (every entry cites a PR #),
     and any breaking change to import/commit behavior must appear here.
  3. **GitHub Releases** page mirrors the CHANGELOG (`https://github.com/embedpdf/embed-pdf-viewer/releases`).
- **Public-API fallback signal for consumers**: there is **no third signal**. Embedpdf
  does not expose "did this `commit()` actually flip anything" — `commit()` returns
  `PdfTaskHelper.resolve(true)` either when there is nothing to commit (line 6823-6824)
  or when `acquireCommitLock` returns false during a concurrent commit (line 6826-6827).
  Consumers cannot distinguish "auto-commit did the work" from "auto-commit happened in
  parallel and resolved to `true` immediately" without reading the state.
- **Could `commit()` silently become necessary again on upgrade?** Yes. If a future
  release deletes the `autoCommit !== false` guard at line 5268 (or sets the default to
  `false`), the `'new' → 'synced'` flip would require an explicit `commit()` call. The
  current `acquireCommitLock` short-circuit (`index.js:6807-6820`) means even then the
  manual call is harmless (resolves to `true` when nothing pending). **So even in the
  worst case, retaining the manual `commit()` is a no-op safety belt, not a bug fix.**
- **Recommendation**: prefer **REMOVE** for code clarity today, and add a one-line
  upstream-pin surveillance note (per §6) so future upgrades that change the default
  config force a deliberate re-audit rather than a silent regression.

### §6 Proposed spec §5.3 sentence (the one-liner for issue #82)

If issue #82's ADR redline decides to drop the manual `commit()` calls, the spec §5.3
sentence should be:

> Under the currently pinned `@embedpdf/plugin-annotation@2.14.4` with the default
> `autoCommit: true` config, `importAnnotations(...)` already dispatches an internal
> `commit()` (processImportItems → `commit(documentId)` → `executeCommitBatch` →
> `commitPendingChanges` reducer), so the explicit `commit()` call that preceded this
> audit is redundant; a future upstream change to the `autoCommit` default or to the
> `processImportItems` guard is the only scenario in which the explicit call becomes
> load-bearing again, and the changelog at
> `https://github.com/embedpdf/embed-pdf-viewer/blob/main/packages/plugin-annotation/CHANGELOG.md`
> is the surveillance signal.

This sentence is single-paragraph, meets the "slottable into spec §5.3" requirement,
and embeds both the REMOVE rationale and the upgrade alarm route.

## Specific edits recommended

1. **`frontend/src/pages/PdfViewer.tsx:495-496`** — DELETE the entire inline comment
   block. The claim "autoCommit:true 只对 CREATE_ANNOTATION 路径生效,import 路径
   不走那条 reducer" is false. Delete the two lines and the `ann.forDocument(docId).commit()`
   call at line 497 entirely. Same edit at lines 542-543.
2. **Issue #82 ADR redline** — Rewrite ADR-0006 pitfall #3 ("commit() required") to
   match the new contract:
   - Old: "在 `importAnnotations` 之后,显式调一次 `commit()`,确保 'new' → 'synced'
     状态 flip 发生在 React 渲染到 PDF 之前"
   - New: "Default `autoCommit: true` makes `importAnnotations` self-committing.
     Do NOT call `commit()` explicitly after import — the auto-commit's `commit()`
     has the same effect. If a future embedpdf upgrade reverses the
     `processImportItems` auto-commit guard, this pitfall becomes load-bearing
     again — the upstream CHANGELOG is the surveillance surface."
3. **`docs/specs/embedpdf-annotation-async-contract.md` §7 (lines 461-488)** — The
   "belt-and-suspenders" framing should be tightened. Today it says the manual call
   is "not equivalent to a no-op" because of pathological timing via
   `acquireCommitLock`. With REMOVE, the manual call no longer exists; the framing
   becomes: "the manual `commit()` call was redundant; auto-commit inside
   `processImportItems` handles the flush, and the `acquireCommitLock` short-circuit
   ensures any manual call would have been a no-op anyway."
4. **`docs/specs/pdf-viewer-url-contract-verification.md` §3 (lines 80-84)** — The
   "NEEDS FIX" wording on §3 step 9 needs to be rephrased: the explicit `commit()` call
   is no longer in the code, so it cannot be a "design intent" or a "bug under
   investigation". The new wording: "Step 9 was `ann.forDocument(docId).commit()` after
   `importAnnotations`. Per `docs/specs/embedpdf-commit-audit.md` (issue #81), the
   manual `commit()` call has been removed in #82: auto-commit-on-import in
   `processImportItems` is structurally sufficient under the pinned v2.14.4 with
   `autoCommit: true`."
