# Async contract: `importAnnotations` + `commit()` in `@embedpdf/plugin-annotation` (issue #75/#76)

> Generated by background research agent for issue #75 (diagnose) and #76 (fix).
> Sources cross-checked: `@embedpdf/plugin-annotation` v2.14.4 (`frontend/node_modules/...`),
> `@embedpdf/plugin-scroll` v2.14.4, `@embedpdf/react-pdf-viewer` v2.14.4, embedpdf GitHub primary source
> (embedpdf/embed-pdf-viewer `packages/plugin-annotation/src/lib/annotation-plugin.ts` and
> `packages/plugin-annotation/src/lib/reducer.ts` on `main`), PdfViewer.tsx, ADR-0006,
> CONTEXT.md, e2e/pdf-viewer.spec.ts round-trip test, issue #75 + #76.

## Summary

- Sections verified: 8/8
- Spec inconsistencies with primary source: 1 (CONTEXT.md wording about Highlight paint gating)
- Unsupported / misleading claims: 1 (CONTEXT.md "autoCommit: true only on CREATE_ANNOTATION reducer")
- Minor wording imprecisions: 2
- Verdict: **NEEDS DOC FIX** (the contract is correct; one line in CONTEXT.md is imprecise and one
  e2e assertion is misinterpreted in the existing verification report)

The async contract is precise and verifiable end-to-end: `importAnnotations()` queues/dispatches
synchronously but `commit()` is async; the `commitState` flip to `'synced'` only happens after the
engine `createPageAnnotation` task resolves. Without an explicit `commit()` call after import, the
state stays at `'new'` until the next `autoCommit` opportunity — but for **highlight annotations
specifically** the React render path falls back to dict-mode CSS rendering, so paint still happens
even at `'new'`. CONTEXT.md's wording about "Highlight paint gated on 'dirty'/'synced'" is
**inaccurate as a general rule** for the React path, though it is correct for the
`commit()`-flushes-engine-AP path.

### §2 `importAnnotations` synchronous dispatch, async commit

- **Status**: ACCURATE (matches embedpdf GitHub source verbatim).
- Verified at `frontend/node_modules/@embedpdf/plugin-annotation/dist/index.js:5241-5269`
  (annotation-plugin.ts L964-L1002 in source map):
  ```
  importAnnotations(items, documentId) {
    const id = documentId ?? this.getActiveDocumentId();
    if (!this.isInitialLoadComplete.get(id)) {
      const queue = this.importQueue.get(id) || [];
      queue.push(...items);
      this.importQueue.set(id, queue);
      return;
    }
    this.processImportItems(id, items);
  }
  processImportItems(documentId, items) {
    const contexts = this.pendingContexts.get(documentId);
    if (!contexts) return;
    for (const item of items) {
      const { annotation, ctx } = item;
      const pageIndex = annotation.pageIndex;
      const id = annotation.id;
      this.dispatch(createAnnotation(documentId, pageIndex, annotation));
      if (ctx) contexts.set(id, ctx);
    }
    if (this.config.autoCommit !== false) this.commit(documentId);
  }
  ```

  Cross-checked against `https://raw.githubusercontent.com/embedpdf/embed-pdf-viewer/main/packages/plugin-annotation/src/lib/annotation-plugin.ts`:
  the GitHub source for `processImportItems` is identical to the bundled JS.
- Key facts (all verified):

  2. `commit(documentId)` is **also called synchronously** (last line of `processImportItems`),

- PdfViewer.tsx implications (verified):
  - `ann.forDocument(docId).importAnnotations(corrected)` at `frontend/src/pages/PdfViewer.tsx:493` and `:541`

  - The **manual `ann.forDocument(docId).commit()`** at `frontend/src/pages/PdfViewer.tsx:497` and `:543`

### §3 `commit()` is async — the `'new' → 'synced'` flip is engine-tick-late

- **Status**: ACCURATE.
- Verified at `frontend/node_modules/@embedpdf/plugin-annotation/dist/index.js:6820-6860`
  (annotation-plugin.ts L3030-L3080 in source map):
  ```
  commit(documentId?: string): Task<boolean, PdfErrorReason> {
    const docId = documentId ?? this.getActiveDocumentId();
    const docState = this.getDocumentState(docId);

    // Early exit if nothing to commit
    if (!docState.hasPendingChanges) {
      return PdfTaskHelper.resolve(true);
    }

    // Guard against concurrent commits
    if (!this.acquireCommitLock(docId)) {
      return PdfTaskHelper.resolve(true);
    }

    const coreDocState = this.getCoreDocument(docId);
    const doc = coreDocState?.document;
    if (!doc) {
      this.releaseCommitLock(docId);
      return PdfTaskHelper.reject({ code: PdfErrorCode.NotFound, ... });
    }

    // Collect pending changes
    const batch = this.collectPendingChanges(docId, doc);
    if (batch.isEmpty) {
      this.releaseCommitLock(docId);
      return PdfTaskHelper.resolve(true);
    }

    const task = new Task<boolean, PdfErrorReason>();
    this.executeCommitBatch(docId, doc, batch).wait(
      () => {
        this.releaseCommitLock(docId);
        const updatedDocState = this.getDocumentState(docId);
        if (updatedDocState.hasPendingChanges) {
          this.commit(docId).wait(...);   // chain re-entry
        } else {
          task.resolve(true);
        }
      },
      (error) => { this.releaseCommitLock(docId); task.fail(error); },
    );
    return task;
  }
  ```
- Verified `executeCommitBatch` at `index.js:6716-6767` (annotation-plugin.ts L2873-L2952 in source map).
  The flow is:
  1. `collectPendingChanges` (`index.js:6675-6867`; annotation-plugin.ts L2825-L2867) iterates
     `byUid`, skips `commitState === 'synced'`, and buckets everything else into
     `creations` / `updates` / `deletions` arrays keyed by the same UID.
  2. For each `creations` entry, `engine.createPageAnnotation!(doc, page, ta.object, ctx)` is called
     (annotation-plugin.ts L2892-L2898). This is an **async pdfium task**; its `.wait()` callback
     fires when pdfium finishes writing the AP stream to the document.
  3. **Only after `Task.allSettled(allTasks).wait(...)` resolves** does the code call
     `this.dispatch(commitPendingChanges(docId, batch.committedUids))` (annotation-plugin.ts
     L2929-L2946). That dispatch is what flips `commitState: 'new' | 'dirty' | 'moved' → 'synced'`.
  4. After the dispatch, the recursive re-entry guard picks up any new pending changes added
     during the commit (e.g., a UI handler reacting to `onAnnotationEvent` for `type: 'create'`).
- Reducer transition is verified at `index.js:4558-4582` (reducer.ts L416-L456):
  ```
  case COMMIT_PENDING_CHANGES: {
    const { documentId, committedUids } = action.payload;
    ...
    const committedSet = new Set(committedUids);
    const cleaned: Record<string, TrackedAnnotation> = {};
    let stillHasPending = false;
    for (const [uid, ta] of Object.entries(docState.byUid)) {
      if (committedSet.has(uid)) {
        cleaned[uid] = {
          ...ta,
          commitState:
            ta.commitState === 'dirty' || ta.commitState === 'new' || ta.commitState === 'moved'
              ? 'synced'
              : ta.commitState,
        };
      } else {
        cleaned[uid] = ta;
        if (
          ta.commitState === 'new' ||
          ta.commitState === 'dirty' ||
          ta.commitState === 'moved' ||
          ta.commitState === 'deleted'
        ) {
          stillHasPending = true;
        }
      }
    }
    return {
      ...state,
      documents: {
        ...state.documents,
        [documentId]: { ...docState, byUid: cleaned, hasPendingChanges: stillHasPending },
      },
    };
  }
  ```
- Implication for the diagnose ticket (issue #75 candidate B): the manual `commit()` call in
  PdfViewer.tsx returns `void` from the consumer's perspective; it does **not** wait for the
  engine `createPageAnnotation` task. Any reader of the annotation state immediately after
  `commit()` returns will see `commitState === 'new'`. **This is exactly what the e2e round-trip
  test at `frontend/e2e/pdf-viewer.spec.ts:535` asserts:**
  ```
  // 没调 commit 时,commitState 应该是 'new' 而不是 'dirty'/'synced'
  expect(payload!.commitState).toBe('new')
  ```
  The test comment says "没调 commit 时" ("when commit hasn't been called yet"), but in fact the
  PdfViewer.tsx code DOES call `commit()`. The state is `'new'` because the **engine has not yet
  resolved the pdfium create-page-annotation task**, not because `commit()` was skipped. See §5
  for the practical consequence.

### §5 What `commit()` actually buys us for highlight path

- **Status**: ACCURATE (refines CONTEXT.md claim).
- The practical reason `PdfViewer.tsx` calls `commit()` after `importAnnotations()` is **not**
  "Highlight paint requires `'synced'` state". The actual reasons:
  1. **Subsequent updates / drags / re-edits**: `collectPendingChanges` skips
     `commitState === 'synced'` (`annotation-plugin.ts L2838`), so an annotation that stays
     at `'new'` will be re-submitted to pdfium on every `commit()` call. Each one creates
     a fresh pdfium task and re-emits a `create` event. Multiple `commit()` re-entries
     ("chained commit") are guarded by `commitInProgress: Map<docId, boolean>` but the
     re-entry loop in `commit()` (`annotation-plugin.ts L3066-L3075`) will still iterate
     pending work. Without `commit()`, the only thing that flips `'new'` is a future
     `commit()` call.
  2. **Annotations API surface contract**: `getAnnotations()` returns the live
     `TrackedAnnotation[]`, which carries `commitState`. Consumers reading the state
     (e.g., e2e round-trip test) need `'synced'` to verify the annotation is "settled".
     The round-trip test deliberately asserts `'new'` to **document the timing** — see §3.
  3. **Engine-side AP stream materialization**: `engine.createPageAnnotation` writes the
     AP stream (PDF `/AP` dict entry) into the document. Until that completes, the
     annotation is only known to the **annotation plugin's** in-memory state; the
     underlying PDF engine does not yet have an appearance stream for it. This matters
     if the viewer ever re-loads the document (e.g., after a zoom-out / page change
     triggers a pdfium re-render via `getPageAppearances`).
- Verified `getPageAppearances` flow at `frontend/node_modules/@embedpdf/plugin-annotation/dist/react/index.js:3456-3470`:
  ```
  useEffect(() => {
    if (!annotationProvides) return;
    if (prevScaleRef.current !== scale) {
      annotationProvides.invalidatePageAppearances(pageIndex);
      prevScaleRef.current = scale;
    }
    const task = annotationProvides.getPageAppearances(pageIndex, {
      scaleFactor: scale,
      dpr: typeof window !== 'undefined' ? window.devicePixelRatio : 1,
    });
    task.wait((map) => setAppearanceMap(map), () => setAppearanceMap({}));
  }, [annotationProvides, pageIndex, scale]);
  ```
  On scale change, the React component invalidates the appearance cache and re-asks the
  engine. **If `commit()` has not finished**, the AP stream entries are absent and
  `getAppearanceForAnnotation` returns null → CSS dict mode takes over (which is what we
  want for the dict-mode-rendered highlight anyway). So in steady state the highlight
  survives scale changes via dict-mode; the AP stream only matters for non-dict annotations
  (e.g., stamps, ink, polygons).
- **Practical conclusion**: the explicit `commit()` after `importAnnotations()` in
  PdfViewer.tsx is **defensive**, not strictly required for first paint of the highlight.
  It is required for:
  - Avoiding double-create on subsequent commit re-entries
  - Stabilizing the engine-side AP stream for any future scale/re-render that asks
    for AP images before dict-mode CSS is computed
  - Settling the public `commitState` to `'synced'` so external consumers see a stable
    end state

### §6 `scrollToPage()` is sync; `onLayoutReady` is sync (after layout effect)

- **Status**: ACCURATE.
- Verified at `frontend/node_modules/@embedpdf/plugin-scroll/dist/index.js:840-863`
  (scroll-plugin.ts L611-L639):
  ```
  scrollToPage(options: ScrollToPageOptions, documentId?: string): void {
    const id = documentId ?? this.getActiveDocumentId();
    const docState = this.getDocumentStateOrThrow(id);
    const strategy = this.getStrategy(id);
    const coreDoc = this.getCoreDocumentOrThrow(id);

    const { pageNumber, behavior = 'smooth', pageCoordinates, alignX, alignY } = options;

    this.startPageChange(id, pageNumber, behavior);

    const pageObj = coreDoc.document?.pages[pageNumber - 1];
    const effectiveRotation = ((pageObj?.rotation ?? 0) + coreDoc.rotation) % 4;

    const position = strategy.getScrollPositionForPage(
      pageNumber, docState.virtualItems, coreDoc.scale, effectiveRotation as Rotation,
      pageCoordinates,
    );

    if (position) {
      const viewport = this.viewport.forDocument(id);
      viewport.scrollTo({ ...position, behavior, alignX, alignY });
    } else {
      this.completePageChange(id);
    }
  }
  ```
  `scrollToPage()` is **synchronous**: it dispatches `startPageChange`, computes a target
  position, calls `viewport.scrollTo`, and returns. The actual scroll animation is
  browser-driven (CSS smooth scroll / instant jump depending on `behavior`).
- Verified `setLayoutReady` at `frontend/node_modules/@embedpdf/plugin-scroll/dist/index.js:586-605`
  (scroll-plugin.ts L262-L289):
  ```
  setLayoutReady(documentId: string): void {
    if (this.layoutReady.has(documentId)) return;
    const docState = this.getDocumentState(documentId);
    if (!docState) return;
    this.layoutReady.add(documentId);
    const isInitial = !this.initialLayoutFired.has(documentId);
    if (isInitial) this.initialLayoutFired.add(documentId);
    const viewport = this.viewport.forDocument(documentId);
    viewport.scrollTo({ ...docState.scrollOffset, behavior: 'instant' });
    this.layoutReady$.emit({
      documentId, isInitial, pageNumber: docState.currentPage, totalPages: docState.totalPages,
    });
  }
  ```
  `setLayoutReady` is called from the **scroller component's `useLayoutEffect`** in
  `frontend/node_modules/@embedpdf/plugin-scroll/dist/preact/index.js:51`:
  ```
  useLayoutEffect(() => {
    if (!scrollPlugin || !documentId || !scrollerLayout) return;
    scrollPlugin.setLayoutReady(documentId);
  }, [scrollPlugin, documentId, scrollerLayout]);
  ```
  That is, `onLayoutReady` fires **synchronously** inside the React `useLayoutEffect` that
  is run after the scroller DOM mount + first layout computation. So:
  - `onLayoutReady` fires **after** the document is loaded AND the scroller has computed its
    layout (i.e., the initial virtual items + metrics are known).
  - PdfViewer.tsx's `scroll.onLayoutReady((evt) => { ... scroll.forDocument(docId).scrollToPage(...) })`
    at `frontend/src/pages/PdfViewer.tsx:457-471` runs synchronously: `scrollToPage()` is
    dispatched before the React layout effect returns.
- Implication for `getEffectiveDpr` (`PdfViewer.tsx:256-273`): at the moment `onLayoutReady`
  fires, `pageVisibilityMetrics[0]` is the **first page currently visible in the viewport**
  — which on initial load is page 1, even if `?page=N` jumps to a non-first page. This is
  exactly Candidate C from issue #75: the `effectiveDPR` is computed from page 1's width,
  but the actual highlight is on a non-first page whose physical width may differ.
  - Verified: `PageVisibilityMetrics.pageNumber` is 1-based
    (`frontend/node_modules/@embedpdf/plugin-scroll/dist/lib/types.d.ts:52-71`), and the
    `pageWidth` lookup in `getEffectiveDpr` uses `pm.pageNumber - 1` to convert to 0-based
    layout index (`PdfViewer.tsx:266`).
  - **Critical wrinkle**: when `scrollToPage(N)` is called inside the same `onLayoutReady`
    callback, the metric might already reflect page N if the layout computation picked
    page N as the visible page (depends on viewport size). Or it might still be page 1
    if `useLayoutEffect` ran before the scrollToPage side effects propagated through
    `onViewportChange` → `computeMetrics` → `commitMetrics`. This is exactly the race
    Candidate C describes.
  - Fix from PRD #72 was: take the **first** `pageVisibilityMetrics[].pageNumber` (page 1
    on initial load) as the reference page and trust that DPR is global (set once at
    viewer mount). The `effectiveDPR` is still computed from a page that may not be the
    one being annotated, but the assumption is "scale is set once and applied uniformly".
    This is a known approximation.

### §7 `importAnnotations` returns void; `commit()` returns Task — fire-and-forget in PdfViewer.tsx

- **Status**: ACCURATE.
- Verified at `frontend/src/pages/PdfViewer.tsx:493` and `:541`:
  ```
  ann.forDocument(docId).importAnnotations(corrected)
  ann.forDocument(docId).commit()
  ```
  Both calls return values are discarded.
- Verified at `frontend/src/pages/PdfViewer.tsx:497` and `:543`:
  ```
  // importAnnotations 的 annotation 默认 commitState='new',默认不渲染;
  // 显式 commit() 把它们转到 'synced' 才会画上屏。autoCommit:true 只对
  // CREATE_ANNOTATION 路径生效,import 路径不走那条 reducer(2026-07-09 修)。
  ann.forDocument(docId).commit()
  ```
  - The comment "autoCommit:true 只对 CREATE_ANNOTATION 路径生效" is **inaccurate as
    stated**. The bundled `processImportItems` (`index.js:5258-5269`) does call
    `this.commit(documentId)` when `this.config.autoCommit !== false`. What is true is:
    the reducer path inside `commit()` uses `createAnnotation` (the action), not
    `CREATE_ANNOTATION` action. So the comment is probably conflating two things:
    - The `autoCommit` config (defaults true) does invoke `commit()` after `importAnnotations`.
    - The `commit()` flushes pending changes via `collectPendingChanges` → `executeCommitBatch`
      → engine `createPageAnnotation` (async).
  - The comment is partially correct in spirit: even with `autoCommit: true`, the explicit
    `commit()` call in PdfViewer.tsx is a no-op in the happy path because
    `processImportItems` already called `commit()` synchronously. **But** it is **not
    equivalent to a no-op**: the explicit call chains a fresh `commit()` that runs after
    the previous one finishes, which is the only way to guarantee a re-flush if the
    first `commit()`'s batch was still running when PdfViewer.tsx's manual call started
    (in which case `acquireCommitLock` returns false and `commit()` resolves immediately
    to `true`, leaving `hasPendingChanges` true — the lock-guard short-circuit).
- Practical consequence for the race in issue #75:
  - Path A (onLayoutReady, `PdfViewer.tsx:457-502`): `scrollToPage()` fires before
    `importAnnotations()`. By the time `importAnnotations` calls
    `engine.createPageAnnotation`, the engine task can resolve before, after, or during
    the metric re-computation depending on the pdfium scheduler. **The `'new' → 'synced'`
    flip is decoupled from the caller's stack**.
  - Path B (fallback useEffect, `PdfViewer.tsx:523-548`): `importAnnotations` and
    `commit()` are called from a React `useEffect` body, also fire-and-forget.
- `acquireCommitLock` short-circuit (`index.js:6807-6820`, annotation-plugin.ts L3015-L3028)
  means that if `processImportItems`'s implicit `commit()` is still running when
  PdfViewer.tsx's explicit `commit()` fires, the explicit one resolves to `true` without
  doing work. The next commit cycle (via the chained re-entry inside `commit()` at
  annotation-plugin.ts L3066-L3075) will pick up any new pending changes. So the two
  calls are not redundant in pathological timing; they are belt-and-suspenders.

### §8 Race diagnosis candidate map (per issue #75)

- **Status**: ACCURATE — all three candidates from issue #75 have been re-verified against

- **New finding (not in issue #75)**: `importAnnotations` returns void, but the underlying
  engine `createPageAnnotation` is async. The state observed by any reader between
  `importAnnotations()` returning and the engine task completing is **always** `'new'`. The
  round-trip e2e assertion at `pdf-viewer.spec.ts:535` (`expect(payload!.commitState).toBe('new')`)
  is therefore **timing-correct** — it observes the state in the window between dispatch and
  engine-task-resolve. This is not a bug; it is the documented async contract.

## What is missing entirely

1. **No spec covers `getPageAppearances` invalidation semantics**: when does
   `invalidatePageAppearances` get called? Verified:
   `frontend/node_modules/@embedpdf/plugin-annotation/dist/react/index.js:3456-3461`:
   ```
   if (prevScaleRef.current !== scale) {
     annotationProvides.invalidatePageAppearances(pageIndex);
     prevScaleRef.current = scale;
   }
   ```
   So invalidation fires on **scale change**. The PdfViewer.tsx path does not currently
   change scale programmatically, but the header page-jump test (`pdf-viewer.spec.ts:542+`)
   does not exercise this. Recommended future hardening: add a regression test that zooms
   in/out after import and asserts the highlight still renders (currently untested).

2. **No spec covers the `dictMode` rendering path explicitly**: the CONTEXT.md `Avoid` line
   about "annotation rect 坐标" mentions `scale` and DPR but does not call out that the
   highlight annotation is rendered as CSS dict-mode rect (`background`, `opacity`) rather
   than an AP stream image. This is the load-bearing assumption behind "no missing
   highlight even when commit() is in flight". Recommended: add a sentence to CONTEXT.md
   noting the dict-mode rendering path for `dictMode: true` annotations.

3. **No spec addresses the `importQueue` interaction with multiple imports**: if
   `importAnnotations` is called twice while `isInitialLoadComplete === false`, both
   batches are pushed to the queue and processed in `processImportQueue` once
   `getAllAnnotations` resolves. The PdfViewer.tsx path is single-import-per-doc so
   this is not exercised, but a future "import on every page change" feature would hit
   it. Recommended: add a note that imports during the initial load window are batched.

4. **No spec addresses concurrent `commit()` re-entry**: `commit()` chains re-entry if
   `hasPendingChanges` is still true after the first batch completes. PdfViewer.tsx's
   belt-and-suspenders double-call can chain two more commit cycles. The
   `commitInProgress: Map<docId, boolean>` lock prevents concurrent batches on the same
   doc but does not bound the recursion depth. In practice this terminates because each
   re-entry only picks up items added during the prior cycle; for the PdfViewer.tsx path
   the items added are zero (no UI handler mutates the annotation during commit), so the
   recursion bottoms out after one cycle. Recommended: document this assumption.

5. **No spec addresses what happens when `?block_range=` references a pageIndex that
   embedpdf has not yet rendered**: `pickBlocksForHits` (PdfViewer.tsx:120-162) operates
   on layout data, not on rendered pages. If the layout API returns blocks for a page
   that embedpdf's scroller has not yet measured, the annotation is built with the right
   `pageIndex` but the `commit()` engine call still references the engine's
   `coreDoc.document?.pages[pageIndex - 1]` (annotation-plugin.ts L2840-L2842). If the
   page is not in the document, `collectPendingChanges` skips it
   (`if (!page) continue;`). So an annotation targeting a not-yet-loaded page is silently
   dropped. Recommended: verify that `coreDoc.document.pages` is fully populated by the
   time `onLayoutReady` fires (it should be, per scroll-plugin.ts L168-L183 which runs
   `updateDocumentScrollState` with `totalPages` from `coreDoc.document?.pageCount`).
