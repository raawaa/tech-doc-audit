## Specific edits recommended

1. **§1.1 lines 25-26 — fix "scrollToPage is never called with NaN"**:
   - Current: `Out of contract: ?page= value ≤ 0 or non-numeric — parseInt returns NaN; scrollToPage is never called with NaN; behavior is undefined, not "silently fall back to page 1". Tests must not rely on that.`
   - Suggested: `Out of contract: ?page= value ≤ 0 or non-numeric — parseInt returns NaN; the resulting `target = NaN` propagates into `scrollToPage({pageNumber: NaN, ...})` (line 464) and embedpdf ignores it. No silent fallback to page 1. Tests must not rely on this.`

2. ~~**§3 step 9 (lines 123-125) + §5.3 (lines 183-185) — soften the "commit flips state" claim to acknowledge the bug under investigation**~~ — **implemented by REMOVE in #82** (both restore paths now call only `importAnnotations(...)`; ADR-0006 pitfall #3 deleted; spec §3 step 9 / §5.3 rewritten to describe the auto-commit-on-import contract).

3. **§9.1 line 279 — remove or qualify the `getRectPositionForPage` reference**:
   - Current: `For new assertions on rect, use \`getRectPositionForPage(rect)\` if visual pixel positions are needed (per CONTEXT.md §"E2E annotation 断言钩子"). Pure data assertions should stick to \`getFirstAnnotationPayload\`.`
   - Suggested: `For new assertions on rect, prefer \`getFirstAnnotationPayload\` (e2e:104) which reads \`rect.origin\` / \`rect.size\` from the JS annotation state. (\`getRectPositionForPage\` is referenced in CONTEXT.md as the visual-pixel-position hook but is not yet defined in the codebase.)`

## What is missing entirely

1. **No spec assertion covers `?page=N` alone (no `block_range` / `?highlight`)** as a first-class use case. The spec discusses it implicitly (§1.1 "Use case: when no block_range / highlight is given, this is the only navigation hint") but does not pin it as a contract item. The round-trip e2e test (line 510) and the header page-jump test (line 542) cover it implicitly. **Recommended**: add a sentence to §1.1 explicitly stating that `?page=N` alone is a valid URL form, used when the auditor manually opens to a specific page (no highlight required).

2. **No spec assertion explicitly covers `?page=N&block_range=A,B` with the A,B range out of bounds for page N** — e.g., `?page=1&block_range=20,30` when only 5 blocks exist on page 1. The current code (`tryBlockRange` returns false → fallback to `?highlight=`) handles this gracefully, but no test exercises it. **Recommended**: either add to §1.2 a "fallback semantics" paragraph or add an e2e test.

3. **No spec assertion covers URL with NO recognized params** (e.g., just `/pdf-viewer/:docId`). The reader has to infer from §1.1 that defaults apply. **Recommended**: add a single sentence to §1 stating "If none of the three parameters is present, the viewer renders the document starting at page 1 with no highlights."

4. **Spec does not address `?page=` with a value exceeding `meta.page_count`**. Since `targetPage` is not bounds-checked against `meta.page_count` anywhere in the code, embedpdf will get a NaN-or-OOB pageNumber. **Recommended**: either document this as "Out of contract: ?page= > meta.page_count" or note that bounds checking is delegated to embedpdf.

5. **Spec does not pin `?highlight=` precedence when only `?highlight=` is provided with NO `?page=`**. The default `targetPage=1` is implicit. This is correct behavior but not stated.

None of these missing items contradict issue #74's deliverable list (which is fully covered); they are hardenings for future-proofing.
