# 复盘 #4：Unicode 文本归一化——从 PDF 高亮定位的匹配失败说起

## 写作背景

最近在做 PDF 文档的引用高亮功能：用户在搜索框输入一段文字，系统要在 PDF 原文中标出对应的位置。这项功能涉及大量字符串匹配，匹配质量直接影响定位准确性。

在开发过程中，字符串比较出现了若干意料之外的失败：用户输入与 PDF 文本看起来一致，`===` 比较却返回 false。排查后定位到 Unicode 文本归一化的问题，由此接触到 NFKC 这一概念。这篇文章整理 NFKC 的原理及其在文本匹配场景中的应用。

## NFKC 解决什么问题

文本匹配是应用开发中的常见需求：搜索查询、字符串对比、查重、关联匹配。匹配失败的原因通常不是算法，而是字符串"看起来一样但底层不等"。

Unicode 标准中，同一个视觉字符可能有多种表示方式，对应不同的 code point。例如，全角数字 ８ 与半角数字 8 在显示上完全一致，但在计算机中的编码不同。归一化（normalization）的作用，就是把这些"视觉等价"的表示统一到一种标准形式，使后续的字符串比较、搜索、匹配能够按预期工作。NFKC 是 Unicode 定义的四种归一化形式之一，是文本匹配场景下的首选。

## 什么是 Unicode 归一化

计算机中每个字符对应一个数字编码，称为 code point。例如，大写字母 A 对应 U+0041。问题在于，同一个"视觉字符"可能对应多个 code point。

常见的等价表示差异包括：

- **全角与半角字符**：全角 ８（U+FF18）与半角 8（U+0038）视觉相同，但 code point 不同
- **兼容性字符**：连字 ﬁ（U+FB01）是拉丁字母 f 和 i 的合字，与两个独立的字符 f、i 在视觉上等价
- **组合与分解**：某些带音标的字符可以用单一 code point 表示（如 é = U+00E9），也可以用基础字符加组合标记表示（e + ́ = U+0065 U+0301）

归一化的目标，就是消除这些等价表示之间的差异，让同一含义的文本在比较时能够匹配。

## 四种归一化形式的命名规则

四种归一化形式 NFC、NFD、NFKC、NFKD 的名称，由五个字母的排列组合构成。理解每个字母的含义，就能推理出每种形式的行为。

| 字母 | 含义 | 行为 |
|---|---|---|
| N | Normalization | 归一化（前缀，标识这是一个归一化形式）|
| **C** | **C**omposition | 组合：尽量用单一 code point 表示字符 |
| **D** | **D**ecomposition | 分解：将字符拆为基础字符与组合标记 |
| **K** | **K**ompatibility | 兼容性合并：把视觉等价但 code point 不同的字符进一步合并 |

五个字母中，C 与 D 互斥（二选一），K 可出现或不出现。因此四个字母的排列共有 2 × 2 = 4 种，对应四种归一化形式：

| 形式 | 命名来源 | C/D 选择 | K 选择 |
|---|---|---|---|
| NFC | Normalization Canonical Composition | C：组合 | 无 K：不做兼容性合并 |
| NFD | Normalization Canonical Decomposition | D：分解 | 无 K：不做兼容性合并 |
| NFKC | Normalization Kompatibility Composition | C：组合 | 有 K：做兼容性合并 |
| NFKD | Normalization Kompatibility Decomposition | D：分解 | 有 K：做兼容性合并 |

由此可见，C 与 D 决定了"组合与分解"的处理，K 决定了"是否做兼容性合并"。NFC 与 NFKC 在组合行为上一致，但 NFKC 多了兼容性合并；NFC 与 NFD 的差异仅在 C 与 D 的选择。

以三个字符为例，四种形式的结果如下：

| 形式 | é 的结果 | ﬁ 的结果 | ８ 的结果 |
|---|---|---|---|
| NFC | é（U+00E9） | ﬁ（U+FB01） | ８（U+FF18） |
| NFD | e + ́（U+0065 U+0301） | f + i（U+0066 U+0069） | ８（U+FF18） |
| NFKC | é（U+00E9） | f + i（U+0066 U+0069） | 8（U+0038） |
| NFKD | e + ́（U+0065 U+0301） | f + i（U+0066 U+0069） | 8（U+0038） |

表格的读取方式：é 涉及"组合与分解"，C/D 列决定结果；ﬁ 与 ８ 涉及"兼容性合并"，K 决定结果（无 K 时保持原 code point，有 K 时合并为视觉等价的常用形式）。

文本匹配场景下，应当选择 NFKC，因为它在处理组合与分解的同时，完成了兼容性合并。

## NFKC 在文本匹配中的应用

NFKC 适用于以下场景：

- 搜索 query 归一化：用户输入与文档内容的匹配
- 文本查重：判断两段文本是否重复
- 字符串对比：版本号、标识符、文件名的比较
- 关联匹配：不同来源数据的字段对齐

NFKC 不适用于以下场景：

- 显示渲染：归一化会改变字符串的视觉表现，应保留原始字符以维持排版效果
- 密码处理：归一化会改变密码的形态，可能导致验证失败或降低安全性
- 自然语言排序：涉及 locale 规则的排序由 ICU Collator 处理，不属于归一化的范畴

在跨语言文本匹配场景（中日英混排、用户输入不可控）中，NFKC 能够解决大部分"看似相同但不相等"的问题。

## 与其他归一化形式的工程取舍

NFC 不做兼容性合并，在处理全角数字、连字等场景时仍会失配，无法满足文本匹配的需求。NFD 和 NFKD 将字符拆得过细，中文的一个字可能被拆成多个 code point，匹配效率低且无实际意义。

NFKC 在合并等价表示的同时，不过度拆分字符，是匹配场景下的最优选择。实践中，跨语言文本匹配在应用 NFKC 后，大多数"为何对不上"的问题可以得到解决。

## 常见误区

**只归一化一方**：比较字符串 a 和 b 时，只对 a 调用归一化而忽略 b，会导致另一半仍存在差异。正确的做法是双方都进行归一化后再比较。

**用 NFC 凑合**：NFC 不做兼容性合并，在全角字符、兼容性字符等场景下必然失败。

**toLowerCase 之前忘了 normalize**：某些 Unicode 字符的 lowercase 形式与原始形态"看起来不同"但 code point 仍不等。正确顺序是先 normalize 再 lowercase。

## 总结

文本匹配的第一原则是双方归一化后再比较。Unicode 归一化是文本处理的基础设施，理解其原理能够避免大量调试时间。对于涉及 locale 规则的排序与比较，需要进一步了解 ICU Collator 或 JavaScript 的 Intl.Collator API。
